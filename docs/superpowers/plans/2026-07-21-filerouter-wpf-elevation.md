# FileRouter WPF Elevation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Elevate the .NET port of FileRouter into a well-designed WPF application — themed (OS light/dark, WCAG-AA-enforced), MVVM-testable, with a redesigned settings experience, real config-driven hotkeys, an in-app history viewer, and the existing safety invariants intact — then retire the WinForms shell.

**Architecture:** `FileRouter.Core` stays untouched except for additive config keys. A new `src/FileRouter.Wpf` project is built **alongside** the working WinForms app and only replaces it at the final task, so the product stays shippable at every commit. All app logic lives in plain-C# view models and services (no WPF types), tested by a new `tests/FileRouter.Wpf.Tests` project; XAML views are thin bindings. The WebView2 "navigate-away-to-release-the-file-handle" dance, the never-overwrite move semantics, and the network-safe SQLite settings are preserved exactly.

**Tech Stack:** .NET 8, WPF (`net8.0-windows`), Microsoft.Web.WebView2 (WPF control, same package already used), PdfSharp + Microsoft.Data.Sqlite (already in Core), System.Security.Cryptography.ProtectedData (DPAPI, new — Microsoft-owned), xUnit. **No MVVM framework** — hand-rolled `ObservableObject`/`RelayCommand`/`AsyncRelayCommand` (~120 lines) keeps the dependency footprint at the project's stated "small, runtime-already-present" goal.

## Global Constraints

- Baseline is green: 183/183 Core tests, clean build. Every task ends green (`dotnet build` + `dotnet test`) and committed.
- Work happens on branch `wpf-elevation` off `main`.
- Files are only ever **moved**, never deleted or overwritten; taken names get a ` (2)` counter. Nothing in this plan may weaken that.
- SQLite stays `journal_mode=TRUNCATE`, `synchronous=FULL`, `busy_timeout=30000` — never WAL (corrupts over SMB).
- New config keys use the **exact Python key names** (`ui_font_family`, `ui_font_size`, `word_separator`, `saved_passwords`, `unlock_suffix`) so a Python-era config round-trips.
- View models and services must not reference WPF types (`System.Windows.*`) except in `Views/`, `Windows/`, `Theme/ThemeManager`, and service *implementations* that wrap WPF controls. This is what keeps them unit-testable.
- All user-facing copy is plain-English, non-technical ("Set aside for later", "Will be filed as:"), matching the existing apps.
- Every color pairing shipped in the theme must pass WCAG AA 4.5:1, enforced by test.
- WinForms `FileRouter.App` is not modified except for Phase 0 hardening; it is deleted in Task 17.

/// <summary>Save that reports failure instead of crashing the app —
/// a read-only or locked config file must never take the session down.</summary>
public static bool TrySave(Config cfg, string path, out string error)

    try { Save(cfg, path); error = ""; return true; }
    catch (Exception ex) when (ex is IOException or UnauthorizedAccessException

    { error = $"Couldn't save settings to {path}: {ex.Message}"; return false; }
}
